## Part 1: Epic 3 review

### Successes and strengths

1. **Vertical slices held end-to-end** ‚Äî Each story shipped pipeline + persistence + intern UI + tests. Write-first submit (`PENDING` flush ‚Üí `AFTER_COMMIT` ‚Üí async work) matches architecture and keeps the HTTP path fast.

2. **Explicit error taxonomy landed** ‚Äî Story 3.3 maps GitHub 404/403/429, LLM timeout, and parse failures to exact user-visible strings; NFR12‚ÄìNFR14 are traceable in code and tests.

3. **`open-in-view: false` discipline continued** ‚Äî Dedicated repository fetch methods (`findByIdForInternStatusPage`, JSON-only light queries, `LEFT JOIN FETCH` where nullable) avoided repeat LazyInitialization surprises from Epic 2.

4. **Security layering stayed consistent** ‚Äî Method-level `@PreAuthorize`, explicit `/reviews/**` rule for INTERN+ADMIN, 404 vs 403 split documented (SSR vs JSON) ‚Äî intentional, tested.

5. **UX-DR4 / UX-DR12 in code** ‚Äî Intern status card terminal states, resubmit from ERROR/REJECTED, submission history `list-group` on task detail with empty-state copy per AC.

6. **Epic 2 ‚ÄúSpring AI spike‚Äù effectively closed** ‚Äî Structured output via `BeanOutputConverter`, think-strip + fence sanitization, and pipeline integration shipped in 3.1 rather than remaining an abstract prep item.

### Challenges and growth areas

1. **UI regression risk when fragments grow** ‚Äî 3.3 reworked `intern-status-card.html` and briefly dropped `data-review-id` / `data-initial-status`; code review restored polling anchors. *Lesson:* any change to the card root must preserve the polling contract and add a regression check (template or smoke test).

2. **Deferred technical debt accumulated in reviews** ‚Äî Thread-pool rejection leaving rows stuck in `PENDING`, duplicate ownership logic across controllers, null guards on `TaskReview` fields for JSON, stale out-of-order poll responses, repository query naming drift ‚Äî all explicitly deferred. *Risk:* these become production edge cases under load or bad data.

3. **Spec vs implementation: PRG vs ‚Äú202 Accepted‚Äù** ‚Äî `epics.md` still phrases FR15 as 202 + body; canonical implementation is PRG redirect (documented in Story 3.1). *Process:* align `epics.md` with architecture or add a one-line ‚Äúcanonical: PRG‚Äù note to prevent future confusion.

4. **Test realism** ‚Äî `TaskServiceTest` needed a real intern id on stubs to match production queries; good catch in 3.3 review. *Pattern:* when mocking persistence, match key fields the repository uses (`intern_id`, etc.).

5. **Mental model: one DB state during AI work** ‚Äî While GitHub+LLM runs, status stays `PENDING` until `LLM_EVALUATED`; UX labels and spinners must stay aligned with that (not a separate ‚ÄúAI Reviewing‚Äù enum value). Documented in 3.2 dev notes ‚Äî worth carrying into mentor-facing copy in Epic 4.
