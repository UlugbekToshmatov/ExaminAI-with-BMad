## Part 1: Epic 4 review

### Successes and strengths

1. **Mentor scope / admin bypass pattern — clean and consistent** — Story 4.1 put scope logic in the service; Story 4.2 extended it via `assertMentorScopeForReview`. This is the ownership consolidation committed to in the Epic 3 retro (T2) — it landed.

2. **JOIN FETCH discipline held** — Both stories used explicit `JOIN FETCH` strategies (queue: task + intern via `findMentorQueue`; detail: `findByIdForMentorDetail` fetching task, course, task.mentor, intern, and issues). No N+1 regressions. `open-in-view: false` discipline from Epic 3 continued cleanly.

3. **Code review → patch cycle worked as a quality gate** — Both stories had meaningful review findings:
   - 4.1: 400 on invalid params; cross-filtered intern/task dropdowns; row link accessibility; verdict truncation with tooltip; detail stub 404/403 tests
   - 4.2: `aiIssueCard` fragment param alignment (correctness fix); server-side `mentorRemarks` max-length enforcement (security/integrity fix)

4. **Proactive Epic 5 wiring** — `MentorDecisionEvent` record created per architecture contract; `AiReviewNotificationListener.onMentorDecision()` stub added with `AFTER_COMMIT + @Async` pattern. Epic 5.1 has its entry point ready without rework.

5. **Fragment reuse** — `ai-feedback-card`, `mentor-action-panel`, `review-status-badge` fragments are clean, parameterized, and reusable. UX-DR2 and UX-DR3 are fully implemented.

6. **"AI assists, human decides" wording discipline** — "AI Suggestion: APPROVE/REJECT" labeling is correct throughout; the mentor's decision is never visually confused with the AI verdict. Traceable to AC and UX-DR15.

### Challenges and growth areas

1. **AC precision on Thymeleaf fragment signatures** — Story 4.2 initially implemented `aiIssueCard` using a `th:object` parent wrapper rather than the AC-mandated explicit `th:fragment="aiIssueCard(issue)"` direct param. Caught in review, fixed with a patch. Root cause: fragment signature in AC was not verified before implementation. *Lesson:* read the exact fragment contract at task-start, not at review time.

2. **UI validation ≠ server-side validation** — `mentorRemarks` had `maxlength="2000"` on the textarea but no service-layer enforcement. Direct POST bypassed the browser limit. *Lesson:* whenever a form field has a UI length constraint, the service must mirror it — make this a default implementation habit.

3. **Cross-filter matrix complexity underestimated in 4.1** — The intern/task filter dropdowns required 2D cross-filtering: intern options respect status + task filter; task options respect status + intern filter. This was non-trivial and not explicit in the initial AC. Implemented correctly but added unplanned complexity.

### Risks and preparation

- **T1 is now critical path** — The async executor rejection risk (thread pool saturation leaves reviews in PENDING, and now also silently drops AFTER_COMMIT email notifications) is load-bearing for 5.1. NFR15 tolerates SMTP delivery failures but assumes the listener at least executes. A rejected thread never reaches email dispatch. This must be resolved before 5.1 starts.
- **SMTP test strategy** — `@TransactionalEventListener + @Async` email delivery is harder to slice-test than a controller. Decide on `GreenMail` or mock `JavaMailSender` before the story starts, not during review.
- **Admin dashboard query strategy** — Cross-intern, cross-mentor visibility (FR33) mirrors the admin queue scope from 4.1. Extend the same pattern rather than a new query approach.

### Significant discovery assessment

Epic 4 **does not invalidate** the Epic 5 breakdown. The event infrastructure is solid and forward-wired. T1 is the only item upgraded from tracked debt to critical path.

**Epic plan update required: No** — proceed with Epic 5 as planned; resolve T1 before 5.1 starts.
