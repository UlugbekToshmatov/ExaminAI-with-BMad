## Part 1: Epic 1 Review

### Successes & Strengths

1. **Schema completeness as a foundation** — All 5 domain tables (`user_account`, `course`, `task`, `task_review`, `task_review_issue`) created in `001-init-schema.sql` during Story 1.2. This means Epic 2–4 have zero schema migration work to do for the tables they depend on. This was a deliberate forward-looking decision that directly reduces Epic 2 risk.

2. **Code review catching real edge cases** — 9 patches applied across 3 code reviews:
   - Self-deactivation guard (Story 1.3) — would have locked out the admin in production
   - Missing input validation on `UserAccountCreateDto` (Story 1.3) — null fields would have reached the service
   - Deactivate error handling (Story 1.3) — missing try/catch on a controller POST
   - `OLLAMA_BASE_URL` env var not wired into `application.yml` (Story 1.1)
   - `sec:authentication` redundant attribute (Story 1.1)
   - Missing admin-route 403 tests for non-ADMIN roles (Story 1.2)
   - Missing ADMIN→`/mentor/reviews` access test (Story 1.2)
   - `@Valid` + `BindingResult` field-level validation replacing banner error (Story 1.3)
   - `RedirectAttributes` flash attribute for deactivate error feedback (Story 1.3)

3. **Security architecture discipline** — BCrypt-12 enforced via `PasswordEncoder` bean injection (never inline instantiation), `@PreAuthorize` on every controller and service method individually, server-side role enforcement over URL patterns alone. No drift from architecture intent.

4. **Explicit deferred item tracking** — `deferred-work.md` captures 19 items with file references and rationale. This makes the deferred backlog visible and reviewable at each epic boundary rather than silently accumulating.

5. **Test architecture** — `@WebMvcTest` + `@MockBean` for security tests avoids DB dependency while covering all authentication paths. `@ExtendWith(MockitoExtension.class)` for service unit tests. Clean, fast, reproducible.

### Challenges & Growth Areas

1. **Spring Boot version drift** — Architecture specifies Spring Boot 3.4.2. Start.spring.io minimum is 3.5.0; Story 1.1 ran with 3.5.13. The Spring AI artifact name also changed: `spring-ai-ollama-spring-boot-starter` → `spring-ai-starter-model-ollama`. Both were discovered at implementation time and handled correctly, but:
   - Architecture docs are now out of sync with actual version
   - Spring AI 1.0.0 + Boot 3.5.x compatibility for `BeanOutputConverter` (used in Epic 3) is unverified
   - Every future story reading architecture docs will see 3.4.2 and need the same mental correction

2. **Recurring first-pass validation gaps** — All 3 stories had review patches related to validation or error handling. Pattern: story ACs describe happy path with precision, but validation requirements are implicit or underspecified in task lists. As a result, first implementation pass consistently omits input validation and missing-ID error handling.

3. **Absolute `href` paths in Thymeleaf templates** — `user-list.html:10` and `user-form.html:37` use bare `/admin/...` paths instead of `th:href="@{...}"`. This breaks under non-root context path deployment and violates Thymeleaf best practice. Risk: if uncorrected, this pattern will propagate to 10+ Epic 2–4 templates.

4. **No reactivate path for deactivated accounts** — Deactivation is irreversible via UI. During Epic 2 testing, accidentally deactivating the seed mentor account (`mentor`/`mentor123`) would require a direct DB reset to recover. Not a blocker, but an operational friction.

### Key Insights

1. The code review process is working as a quality gate, but the recurring validation pattern suggests story task lists should explicitly include subtasks for input validation and error handling — not just acceptance criteria references.

2. Spring Boot version drift is a low-severity issue for Epics 1–2 (pure Spring MVC + JPA) but becomes a real risk in Epic 3 when Spring AI `BeanOutputConverter` is first invoked. A compatibility spike before Epic 3 begins is strongly recommended.

3. The absolute href pattern in templates is a "copy forward" risk — the next developer creating mentor templates may copy the admin template structure and perpetuate the issue. Fixing it before Epic 2 closes the risk at source.
