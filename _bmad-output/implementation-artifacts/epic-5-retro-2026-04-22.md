## Part 1: Epic 5 review

### Successes and strengths

1. **Event-driven email delivery matches architecture** — `NotificationService` consolidates `AFTER_COMMIT` + `@Async` listeners; duplicate `AiReviewNotificationListener` removed to prevent double-send. NFR15 honored: SMTP failures logged, not rethrown.

2. **Symmetry in defensive guards** — Review feedback added `internId` null guard on mentor-decision path to mirror `mentorId` handling on AI-complete; tests cover skip paths and non-propagating send failures.

3. **Admin dashboard reuses proven patterns** — `JOIN FETCH` on `TaskReviewRepository` for admin rows; filter UX mirrors mentor queue (GET + `onchange`); `review-status-badge` fragment reused; invalid `status` query returns 400 aligned with mentor queue spirit.

4. **Security tests for admin surface** — `@WebMvcTest` with `@WithMockUser` validates 200 for ADMIN and 403 for non-admin on `/admin/dashboard`; integrates with existing security test style.

5. **Scope discipline on 5.2** — Story file explicitly scoped out “consecutive rejections” UI where epics ACs did not require it, reducing gold-plating risk.

### Challenges and growth areas

1. **Epic 4 critical path T1 still open** — `AsyncConfig` does not set a rejection policy (e.g. `CallerRunsPolicy`) or dead-letter logging for saturated pool. Epic 4 retro marked this as blocking for 5.1; 5.1 shipped with mocks proving correctness, but **runtime behavior under pool exhaustion remains a known risk** for notification delivery (and pipeline work competing for the same executor).

2. **SMTP integration testing choice** — 5.1 used mocked `JavaMailSender` (acceptable per story). GreenMail-style integration remains optional; team should decide if Epic 6 or hardening sprint adds full SMTP integration tests.

3. **Empty-state copy vs filtered zero rows** — 5.2 uses single “No submissions yet.” for both no data and filter miss; story flags this as a possible UX follow-up.

4. **Deep link base URL** — 5.1 open question: production links may need `APP_BASE_URL` (or equivalent) if the app is not always same-origin as assumed in emails.

## Part 2: Epic 6 preparation

### Epic 6 preview

**Epic 6: Production Deployment** — multi-stage Dockerfile, `docker-compose.yml` (postgres, ollama, app), healthchecks, named volumes, `.env` injection, graceful async shutdown verification.

| Story | Scope |
|-------|-------|
| 6.1 Production-Ready Docker Compose Deployment | Full stack runnable with one command; persistence; actuator health |

### Dependencies on Epic 5

| Dependency | Readiness |
|------------|-----------|
| `MAIL_*` env vars and notification path | Implemented; compose must pass through env |
| Admin dashboard at `/admin/dashboard` | Implemented; smoke-test in containerized stack valuable |
| Same `AsyncConfig` executor for mail + pipeline | **Risk:** shared pool without rejection policy — load testing or policy fix before declaring production-ready |

### Risks and preparation

- **T1 (executor rejection)** — Epic 6 ACs reference graceful shutdown (`awaitTerminationSeconds: 120`) but not pool saturation. Recommend either implementing rejection handler + observability, or explicitly documenting accepted risk and monitoring plan.
- **Ollama model pull on first start** — Compose story assumes pre-pull; validate timing and disk on CI/developer machines.
- **End-to-end mail in Docker** — Optional Mailpit/MailHog sidecar for local compose validation of FR29/FR30.

### Significant discovery assessment

Epic 5 **does not invalidate** the Epic 6 breakdown. The main adjustment is **operational hardening**: address or formally accept T1 as part of “production-ready” criteria.

**Epic plan update required for Epic 6:** No structural change to stories; **yes** to tightening acceptance around async/email reliability if the team defines production SLOs.
